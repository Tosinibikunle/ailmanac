Это страница, где примитивы перестают быть отдельными функциями и начинают становиться *рабочим процессом*. Если вы уже знаете, что такое субагент или хук, то рычаг — в том, как вы их **объединяете**: слэш-команда, запускающая проход в режиме планирования, субагенты, распределяющие работу веером, хук, который не даст завершить ход, пока тесты не пройдут, и автономный вызов, который выполняет всё это в CI. Давайте соединим это вместе.

<VerifyNote lastVerified="2026-06-24" source="https://code.claude.com/docs/en/best-practices">
Claude Code развивается быстро: флаги, встроенные команды (`/deep-research`, `/code-review`), ключевое слово `ultracode`, доступность динамических рабочих процессов и уровни усилий — всё это меняется между версиями. Количество звёзд у сообществ-репозиториев меняется ежедневно. Сверяйте всё, что зависит от версии, с официальной документацией, прежде чем на это полагаться.
</VerifyNote>

## Ментальная модель: кто держит план?

Каждый продвинутый приём ниже — это разный ответ на один вопрос: *кто держит план и промежуточные результаты?* Собственная формулировка Anthropic для новой функции динамических рабочих процессов чётко это раскладывает:

| Инструмент | Кто решает, что запускается дальше | Где живут результаты | Масштаб |
| --- | --- | --- | --- |
| **[Субагенты](/docs/claude-code/subagents)** | Claude, ход за ходом | Контекстное окно Claude | Несколько за ход |
| **[Навыки](/docs/claude-code/skills)** | Claude, следуя промпту | Контекстное окно Claude | Как и у субагентов |
| **Команды агентов** | Ведущий агент, ход за ходом | Общий список задач | Несколько долгоживущих равноправных |
| **[Динамические рабочие процессы](/docs/claude-code/dynamic-workflows)** | Скрипт | Переменные скрипта | От десятков до сотен |

Прогрессия — это вся суть игры: начните с одного Claude, делегируйте субагентам, чтобы защитить контекст, и переносите план *в код* только тогда, когда задаче нужно больше агентов, чем может скоординировать один разговор.

## Приём 1 — Параллельные субагенты, затем переход к рабочему процессу

[Субагент](/docs/claude-code/subagents) — это отдельный Claude со своим контекстным окном и ограниченным набором инструментов; он возвращает результат, а не свою стенограмму. Привычка опытного пользователя — **распределять веером независимую работу**:

<PromptCard title="Распределить ревью по модулям веером">{`Review the changes in auth/, billing/, and api/ — use the code-reviewer
subagent on each, in parallel. Report only correctness bugs and missing tests.`}</PromptCard>

Три рецензента работают одновременно, каждый тратит свой контекст на диффы, и ваша основная сессия видит три аккуратных отчёта вместо трёх сырых диффов. Подвох: параллелизм помогает только для **независимых** подзадач. Если шагу B нужен результат шага A, выполняйте их последовательно; если они пишут в одни и те же файлы, изолируйте их в [git worktrees](/docs/claude-code/worktrees).

Когда задача перерастает один разговор — миграция на 500 файлов, общекодовый прогон по поиску багов, исследование, перекрёстно проверенное по множеству источников — переходите к **[динамическому рабочему процессу](/docs/claude-code/dynamic-workflows)**. Claude пишет JavaScript-скрипт оркестрации (распределить веером → свернуть → синтезировать), среда выполнения исполняет его в фоне с до 16 параллельными агентами, и в ваш контекст попадает только финальный ответ. Запустите его ключевым словом `ultracode` или просто попросите обычными словами:

<PromptCard title="Запустить динамический рабочий процесс">{`ultracode: audit every API endpoint under src/routes/ for missing auth checks,
then cross-check each finding with a second agent before reporting`}</PromptCard>

Убойная фича — не просто *больше агентов*, а то, что скрипт может применить воспроизводимый **паттерн качества**, например когда независимые агенты состязательно проверяют находки друг друга, прежде чем хоть одна будет доложена. Попробуйте встроенную команду `/deep-research <question>`, чтобы увидеть этот паттерн вживую: она голосует по каждому утверждению и отсеивает те, что не выдерживают перекрёстной проверки.

<Callout type="tip" items={["Сохраните хороший прогон рабочего процесса как команду: откройте /workflows, выберите прогон, нажмите s. Он становится /ваше-имя в каждой будущей сессии.","Оцените стоимость сперва на срезе — одна директория, а не весь репозиторий. Прогон может породить гораздо больше агентов (и токенов), чем один разговор."]} />

## Приём 2 — Хуки: детерминированное связующее звено

Инструкции в CLAUDE.md носят *рекомендательный* характер — Claude обычно им следует. [Хуки](/docs/claude-code/hooks) *детерминированы* — они запускают скрипт в фиксированной точке цикла, гарантированно, каждый раз. Тянитесь к хуку, когда что-то **должно** произойти без единого исключения.

Три хук-паттерна с наибольшим рычагом:

<Steps items={[{title: "PostToolUse — линтинг/форматирование при каждом редактировании", body: "После того как Claude отредактирует файл, автоматически запускайте ваш форматтер или линтер, чтобы кодовая база никогда не дрейфовала. Обратная связь также возвращается к Claude, поэтому он сам себя корректирует."},{title: "Stop — контроль хода на зелёной проверке", body: "Хук Stop запускает ваш скрипт теста/сборки и блокирует завершение хода, пока он не пройдёт. Именно это позволяет автономному прогону завершиться корректно, вместо того чтобы остановиться, когда работа просто 'выглядит готовой'."},{title: "PreToolUse — блокировка запрещённых действий", body: "Запретите записи в защищённый путь (миграции, генерируемые файлы, секреты) до того, как они произойдут, независимо от намерений Claude."}]} />

Вам не нужно вручную писать JSON. Попросите Claude написать хук за вас:

<PromptCard title="Пусть Claude напишет хук">{`Write a hook that runs eslint --fix after every file edit, and a second hook
that blocks any Write or Edit to the db/migrations/ folder. Add them to
.claude/settings.json and show me the config.`}</PromptCard>

<Callout type="warning" items={["Хук Stop, который продолжает блокировать, будет переопределён после нескольких последовательных блокировок, чтобы сессия не зашла в тупик — ваш контроль это защитное ограждение, а не бесконечный цикл.","Хуки запускаются с правами вашей оболочки. Просматривайте любой хук-скрипт перед его коммитом, ровно как вы делали бы с конфигом CI."]} />

## Приём 3 — Кастомные слэш-команды, навыки и стили вывода

Всё, что вы запрашиваете дважды, должно стать примитивом. Решение простое: **навыки — это знание, хуки — это гарантии, MCP — это действие, слэш-команды — это точки входа.**

[Слэш-команда](/docs/claude-code/slash-commands) (или навык с `disable-model-invocation: true`) упаковывает воспроизводимый рабочий процесс, который вы запускаете вручную. `$ARGUMENTS` делает его параметризованным:

<PromptCard title=".claude/skills/fix-issue/SKILL.md">{`---
name: fix-issue
description: Triage and fix a GitHub issue end-to-end
disable-model-invocation: true
---
Fix GitHub issue $ARGUMENTS:
1. gh issue view to read the issue
2. Search the codebase for the relevant files
3. Write a failing test that reproduces the bug
4. Implement the fix, then run tests and lint until green
5. Commit with a descriptive message and open a PR`}</PromptCard>

Запустите её командой `/fix-issue 1234`. Используйте `disable-model-invocation: true` для всего, у чего есть побочные эффекты, что вы хотите запускать осознанно, а не давать Claude тянуться к этому самому.

[Стили вывода](/docs/claude-code/output-styles) меняют то, *как* Claude общается на протяжении всей сессии — лаконично для эксперта, подробно с объяснениями для обучения или в структурированном формате, который может разобрать ваша оснастка. Соедините кастомную слэш-команду (*что*) со стилем вывода (*как*) — и вы оформили оба конца взаимодействия.

Самый недооценённый продвинутый приём из официальной документации: **дайте Claude взять у вас интервью** перед крупной фичей, а затем начните свежую сессию, чтобы строить по записанной спецификации.

<PromptCard title="Спецификация через интервью, затем сборка в чистом контексте">{`I want to build [brief description]. Interview me in detail using the
AskUserQuestion tool — technical implementation, UI/UX, edge cases, tradeoffs.
Dig into the hard parts I might not have considered. When we've covered
everything, write a complete, self-contained spec to SPEC.md.`}</PromptCard>

## Приём 4 — Постройте MCP-стек

[MCP-серверы](/docs/claude-code/mcp) — это исполняемые процессы, которые Claude вызывает по JSON-RPC, чтобы реально *что-то делать* — запрашивать вашу базу данных, читать Sentry, подтягивать дизайн из Figma, заводить задачу в Linear. «Стек» опытного пользователя — это небольшой, продуманный набор, а не всё, что вы смогли найти:

<PromptCard title="Добавьте серверы в ваш стек">{`claude mcp add --transport stdio sentry -- npx -y @sentry/mcp-server
claude mcp add --transport http linear https://mcp.linear.app/sse`}</PromptCard>

Два принципа держат MCP-стек быстрым:

- **Предпочитайте CLI там, где он есть.** `gh`, `aws`, `gcloud` и `sentry-cli` — *самый* контекстно-эффективный способ общаться с сервисом: Claude их уже знает, и они не загружают схему инструментов в каждый ход. Резервируйте MCP для сервисов без хорошего CLI или там, где вам нужен структурированный, типизированный доступ.
- **Держите список серверов лёгким.** Определения инструментов каждого подключённого сервера потребляют контекст заранее. Обрезайте серверы, которыми вы активно не пользуетесь; раздутые списки инструментов вытесняют ваши настоящие инструкции.

Для более глубокого «почему» см. [Инженерия контекста](/docs/frontiers/context-engineering) — та же логика бюджета внимания, что управляет CLAUDE.md, управляет и вашим списком инструментов.

## Приём 5 — Автономные конвейеры и Agent SDK

`claude -p "prompt"` запускает Claude **неинтерактивно** — без сессии, с разбираемым выводом — и это дверь в CI, pre-commit хуки и пакетные задания. Это поверхность [автономного режима / Agent SDK](/docs/claude-code/headless-and-agent-sdk).

Классический паттерн распределения-веером-по-файлам из официального руководства по лучшим практикам:

<PromptCard title="Автономная пакетная миграция">{`# 1. Have Claude generate the work list first, then loop:
for file in $(cat files.txt); do
  claude -p "Migrate $file from React to Vue. Return OK or FAIL." \\
    --allowedTools "Edit,Bash(git commit *)"
done`}</PromptCard>

Два флага делают всю тяжёлую работу: `--output-format json` (или `stream-json --verbose`) делает результаты машиночитаемыми, а `--allowedTools` точно ограничивает, к чему Claude может прикасаться, когда никто не наблюдает. Направляйте это куда угодно:

<PromptCard title="Claude как этап конвейера">{`cat error.log | claude -p "Cluster these errors by root cause, output JSON" \\
  --output-format json | jq '.[] | select(.severity=="high")'`}</PromptCard>

<Callout type="warning" items={["Всегда тестируйте промпт на 2–3 элементах, прежде чем спускать цикл на 2 000. Доработайте, затем масштабируйте.","Автономным прогонам нужна встроенная верификация — хук Stop или шаг с тестом прямо в промпте — иначе вы только что автоматизировали производство правдоподобно-но-неверного вывода в масштабе."]} />

## Приём 6 — Дисциплина режима планирования и инженерия контекста

Самый надёжный рычаг качества — **отделение исследования от исполнения**. Четырёхфазный цикл из руководства Anthropic по лучшим практикам:

<Steps items={[{title: "Исследовать (режим планирования)", body: "Войдите в режим планирования. Claude читает файлы и отвечает на вопросы, но не вносит изменений. Направьте его на точные директории: 'read /src/auth and explain how sessions work.'"},{title: "Планировать", body: "Попросите детальный план реализации. Отредактируйте его прямо перед утверждением — план, который вы исправили, превосходит план, который вы пробежали глазами."},{title: "Реализовать", body: "Выйдите из режима планирования. Claude кодит по плану и запускает указанную вами верификацию."},{title: "Проверить и закоммитить", body: "Пусть субагент со свежим контекстом проверит дифф относительно плана, исправит пробелы, затем закоммитит и откроет PR."}]} />

См. [Режим планирования](/docs/claude-code/plan-mode) для дисциплины и пропускайте его, когда дифф умещается в одно предложение — у планирования есть накладные расходы. Причина, почему это работает, — **экономия контекста**: производительность деградирует по мере заполнения окна, поэтому инструкция на 50 токенов в контексте на 2 000 токенов бьёт куда сильнее, чем та же инструкция, погребённая в 50 000 токенов. Практические привычки:

- `/clear` между несвязанными задачами; чистая сессия с лучшим промптом превосходит длинную загрязнённую.
- После двух неудачных исправлений перестаньте исправлять — `/clear` и перепишите промпт с учётом того, что узнали.
- Делегируйте исследование субагентам, чтобы чтение файлов происходило в *их* контексте, а не в вашем.

## Мастерство CLAUDE.md

[CLAUDE.md](/docs/claude-code/claude-md) загружается в начале каждого разговора, поэтому это самый высокочастотный контекст, который вы контролируете — и потому его легче всего испортить переполнением.

| Класть в CLAUDE.md | Держать снаружи |
| --- | --- |
| Bash-команды, которые Claude не может угадать | Всё, что Claude находит, читая код |
| Правила стиля кода, отличающиеся от дефолтов | Стандартные соглашения, которые Claude уже знает |
| Тест-раннер + как запустить один тест | Полную документацию по API (вместо этого ссылка) |
| Этикет репозитория (именование веток/PR) | Информацию, которая часто меняется |
| Неочевидные подводные камни и причуды окружения | Банальности «пиши чистый код» |

Относитесь к нему как к коду: пересматривайте, когда поведение идёт не так, регулярно отчищайте и используйте импорты `@path/to/file` плюс CLAUDE.md по директориям, чтобы каждая часть монорепозитория получала только то, что релевантно. Переносите *иногда релевантное* знание в [навыки](/docs/claude-code/skills), чтобы оно загружалось по запросу, а не облагало налогом каждый ход.
