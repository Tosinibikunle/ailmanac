Si [construire votre propre boucle d'agent](/docs/api/building-agents) représente plus d'infrastructure que vous ne souhaitez en gérer, un agent **géré** (hébergé par Anthropic) exécute la boucle pour vous — vous vous concentrez ainsi sur le *travail* de l'agent, et non sur la plomberie des sessions, les nouvelles tentatives, l'état et la planification.

## Les deux objets : Agent vs Session

C'est le modèle mental auquel tout le reste se rattache. Ils sont séparés volontairement.

- Un **Agent** est une *configuration persistée et versionnée* — modèle, prompt système, outils, serveurs MCP et compétences (skills). Vous le créez une fois. Chaque mise à jour crée une nouvelle version immuable.
- Une **Session** est une *instance d'exécution* — une seule exécution qui pointe vers un agent par son ID. La configuration vit sur l'agent, jamais sur la session.

```mermaid
flowchart LR
  A[Agent config<br/>versioned] -->|"session pins a version"| S1[Session run]
  A --> S2[Session run]
  A --> S3[Session run]
  S1 --> C[Container<br/>tools execute here]
```

<Callout type="tip">
Les sessions **s'épinglent** à la version de l'agent avec laquelle elles ont été créées : les sessions en cours conservent leur version, les nouvelles sessions obtiennent la plus récente. C'est ainsi que vous déployez des changements de configuration sans casser le travail en cours.
</Callout>

## Ce que le « géré » vous apporte

Plutôt que de bricoler et d'héberger la boucle vous-même, vous obtenez des briques hébergées :

- **Sessions** — des exécutions persistantes que vous créez par exécution et que vous reprenez ; diffusez les événements en flux via SSE.
- **Environnements** — l'infrastructure de conteneurs, soit `cloud` (hébergée par Anthropic) soit `self_hosted` (les outils s'exécutent dans votre propre VPC). Un conteneur par session est l'espace de travail de l'agent.
- **Magasins de mémoire** — un état persistant entre les sessions, avec versionnage et expurgation, sans que vous ayez à câbler une base de données.
- **Coffres-forts (Vaults)** — des secrets pour l'authentification MCP et d'autres services.
- **Déploiements planifiés** — des agents qui s'exécutent selon une planification cron, sans surveillance.

## Coffres-forts : des secrets que le modèle ne voit jamais

Un agent autonome a souvent besoin d'une clé d'API — mais le *modèle* ne devrait jamais la lire. Les identifiants de coffre-fort (`mcp_oauth`, `static_bearer`, `environment_variable`) sont substitués à la sortie (egress) : un identifiant `environment_variable` est injecté dans le bac à sable au moment de l'exécution et n'est *jamais visible* par le modèle.

<Callout type="warning">
C'est le modèle sûr pour donner à un agent un accès puissant. Ne collez pas de clés dans le prompt système ou dans un message — elles deviennent une partie du contexte que le modèle (et vos journaux) peuvent voir. Placez-les dans un coffre-fort.
</Callout>

## Déploiements planifiés : un agent sur un cron

Un **déploiement** attache une planification cron à un agent. Lorsque la planification se déclenche, il démarre une nouvelle session et accomplit sa tâche — aucun planificateur à construire ou à héberger de votre côté. Idéal pour une synchronisation de données nocturne, une analyse de conformité hebdomadaire ou un résumé quotidien.

  {title: "Chaque déclenchement = une exécution", body: "Chaque tentative de déclenchement crée un enregistrement d'exécution (préfixe drun_). Le succès porte un session_id ; l'échec porte un error.type (par ex. environment_archived, session_rate_limited). Listez les exécutions via GET /v1/deployment_runs?deployment_id=..."},
  {title: "Contrôler le cycle de vie", body: "La mise en pause supprime les futurs déclenchements (les exécutions manuelles fonctionnent toujours) ; la reprise repart à la prochaine occurrence et ne rattrape PAS les déclenchements manqués ; l'archivage est définitif."},
  {title: "Déclencher à la demande", body: "POST /v1/deployments/{id}/run démarre une session immédiatement — même en pause — avec trigger_context.type: manual."}

<PromptCard title="Une analyse de conformité hebdomadaire, le vendredi à 20h00 heure de New York">{`POST /v1/deployments
{
  "name": "Weekly compliance scan",
  "agent": "$AGENT_ID",
  "environment_id": "$ENVIRONMENT_ID",
  "initial_events": [
    {"type": "user.message", "content": [{"type": "text", "text": "Run the compliance scan and summarize findings."}]}
  ],
  "schedule": {"type": "cron", "expression": "0 20 * * 5", "timezone": "America/New_York"}
}`}</PromptCard>

<Callout type="tip">
Cron est `minute hour day-of-month month day-of-week`, avec une granularité à la minute. L'heure d'été utilise une sémantique d'horloge murale : une heure qui n'existe pas lors du passage à l'heure d'été est ignorée ; une heure qui se produit deux fois lors du retour à l'heure d'hiver se déclenche deux fois. Choisissez un fuseau horaire et une heure qui évitent ces cas limites pour tout ce qui est sensible.
</Callout>

## Quand choisir le géré vs le personnalisé

| Choisissez le **géré** quand… | Choisissez une **boucle personnalisée / SDK** quand… |
|---|---|
| Vous voulez que l'hébergement, l'état, la planification et les secrets soient pris en charge | Vous avez besoin d'un contrôle total sur la boucle et les outils |
| Vous prototypez rapidement | Vous avez des besoins stricts d'infra/conformité personnalisés |
| La simplicité opérationnelle compte plus que le contrôle | Vous intégrez en profondeur dans votre propre stack |

C'est un spectre — appel unique → workflow → agent personnalisé (SDK) → géré. Commencez aussi simplement que la tâche le permet ; ne montez en complexité que lorsque vous en avez besoin.

## Les mêmes garde-fous s'appliquent

Géré ou non, un agent autonome entreprend tout de même des actions. Conservez le **moindre privilège**, des **coûts/itérations bornés** et l'**approbation humaine pour les étapes risquées** — voir [Sécuriser les agents](/docs/security/securing-agents) et [Renforcer les exécutions autonomes](/docs/security/hardening-autonomous-runs).
