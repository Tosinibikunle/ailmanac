[직접 에이전트 루프를 구축](/docs/api/building-agents)하는 것이 원하는 것보다 더 많은 인프라를 떠맡는 일이라면, **관리형**(Anthropic 호스팅) 에이전트가 루프를 대신 실행해 줍니다 — 그래서 여러분은 세션 배관, 재시도, 상태, 스케줄링이 아니라 에이전트의 *일* 자체에 집중하게 됩니다.

## 두 가지 객체: Agent vs Session

이것이 나머지 모든 것이 매달려 있는 멘탈 모델입니다. 이 둘은 의도적으로 분리되어 있습니다.

- **Agent**는 *영속적이고 버전이 관리되는 구성*입니다 — model, system prompt, tools, MCP 서버, skills. 한 번만 생성합니다. 모든 업데이트는 새로운 불변 버전을 만듭니다.
- **Session**은 *런타임 인스턴스*입니다 — ID로 에이전트를 가리키는 하나의 실행. 구성은 에이전트에 존재하며, 절대 세션에 존재하지 않습니다.

```mermaid
flowchart LR
  A[Agent config<br/>versioned] -->|"session pins a version"| S1[Session run]
  A --> S2[Session run]
  A --> S3[Session run]
  S1 --> C[Container<br/>tools execute here]
```

<Callout type="tip">
세션은 생성될 때의 에이전트 버전에 **고정(pin)**됩니다: 실행 중인 세션은 자신의 버전을 유지하고, 새 세션은 최신 버전을 받습니다. 이것이 진행 중인 작업을 깨뜨리지 않고 구성 변경을 배포하는 방법입니다.
</Callout>

## "관리형"이 제공하는 것

루프를 직접 만들고 호스팅하는 대신, 호스팅된 빌딩 블록을 얻습니다:

- **Sessions** — 실행마다 생성하고 재개하는 영속적인 실행; SSE를 통해 이벤트를 스트리밍합니다.
- **Environments** — 컨테이너 인프라로, `cloud`(Anthropic 호스팅) 또는 `self_hosted`(도구가 여러분 자신의 VPC에서 실행됨) 중 하나입니다. 세션당 하나의 컨테이너가 에이전트의 작업 공간입니다.
- **Memory stores** — 세션 전반에 걸친 영속적 상태로, 버전 관리와 편집(redaction)을 제공하며, 데이터베이스를 직접 연결할 필요가 없습니다.
- **Vaults** — MCP 인증 및 기타 서비스를 위한 비밀 정보.
- **Scheduled deployments** — cron 일정에 따라 무인으로 실행되는 에이전트.

<PromptCard title="에이전트(버전 관리된 구성)를 생성한 다음, 그것에 대해 세션을 실행하기">{`# 1. Create the agent once
POST /v1/agents        -> returns $AGENT_ID
# 2. Each execution is a session pinned to that agent
POST /v1/sessions      { "agent": "$AGENT_ID" }`}</PromptCard>

## Vaults: 모델이 절대 보지 못하는 비밀 정보

자율 에이전트는 종종 API 키가 필요합니다 — 하지만 *모델*은 절대 그것을 읽어서는 안 됩니다. Vault 자격 증명(`mcp_oauth`, `static_bearer`, `environment_variable`)은 egress 시점에 치환됩니다: `environment_variable` 자격 증명은 실행 시점에 샌드박스에 주입되며 모델에게는 *절대 보이지 않습니다*.

<Callout type="warning">
이것이 에이전트에게 강력한 접근 권한을 부여하는 안전한 패턴입니다. 키를 system prompt나 메시지에 붙여넣지 마세요 — 그것들은 모델(그리고 여러분의 로그)이 볼 수 있는 컨텍스트의 일부가 됩니다. vault에 넣으세요.
</Callout>

## 예약 배포: cron 위의 에이전트

**deployment**는 cron 일정을 에이전트에 연결합니다. 일정이 발동되면 새 세션을 시작하고 작업을 완료합니다 — 여러분이 구축하거나 호스팅할 스케줄러가 없습니다. 야간 데이터 동기화, 주간 규정 준수 스캔, 또는 일일 다이제스트에 좋습니다.

<Steps items={[
  {title: "일정 정의하기", body: "agent, environment_id, initial_events(반드시 user.message를 포함해야 함), 그리고 schedule(POSIX cron 표현식과 IANA timezone)을 담아 POST /v1/deployments를 호출합니다."},
  {title: "각 발동 = 한 번의 실행", body: "모든 트리거 시도는 run 레코드(drun_ 접두사)를 생성합니다. 성공은 session_id를 동반하고, 실패는 error.type(예: environment_archived, session_rate_limited)을 동반합니다. GET /v1/deployment_runs?deployment_id=... 로 실행 목록을 조회합니다."},
  {title: "라이프사이클 제어하기", body: "Pause는 향후 트리거를 억제합니다(수동 실행은 여전히 동작); unpause는 다음 발생 시점에 재개되며 놓친 트리거를 소급 실행(backfill)하지 않습니다; archive는 종결 상태입니다."},
  {title: "온디맨드로 트리거하기", body: "POST /v1/deployments/{id}/run은 즉시 세션을 시작합니다 — 일시 중지 중이더라도 — trigger_context.type: manual 로 동작합니다."}
]} />

<PromptCard title="주간 규정 준수 스캔, 뉴욕 시간 매주 금요일 20:00">{`POST /v1/deployments
{
  "name": "Weekly compliance scan",
  "agent": "$AGENT_ID",
  "environment_id": "$ENVIRONMENT_ID",
  "initial_events": [
    {"type": "user.message", "content": [{"type": "text", "text": "Run the compliance scan and summarize findings."}]}
  ],
  "schedule": {"type": "cron", "expression": "0 20 * * 5", "timezone": "America/New_York"}
}`}</PromptCard>

<Callout type="tip">
Cron은 `minute hour day-of-month month day-of-week`이며, 분 단위 정밀도를 가집니다. DST는 벽시계(wall-clock) 의미론을 사용합니다: 봄철 시간 전환에서 존재하지 않는 시간은 건너뛰고, 가을철 시간 전환에서 두 번 발생하는 시간은 두 번 발동됩니다. 민감한 작업이라면 그러한 경계를 피하는 timezone과 시간을 선택하세요.
</Callout>

## 언제 관리형 vs 커스텀을 선택할까

| **관리형**을 선택할 때… | **커스텀 루프 / SDK**를 선택할 때… |
|---|---|
| 호스팅, 상태, 스케줄링, 비밀 정보 처리를 맡기고 싶을 때 | 루프와 도구에 대한 완전한 제어가 필요할 때 |
| 빠르게 프로토타이핑하고 있을 때 | 엄격한 커스텀 인프라/규정 준수 요구사항이 있을 때 |
| 제어보다 운영 단순성이 더 중요할 때 | 자신의 스택에 깊이 임베딩하고 있을 때 |

이것은 스펙트럼입니다 — 단일 호출 → 워크플로 → 커스텀 에이전트(SDK) → 관리형. 작업이 허용하는 만큼 단순하게 시작하세요; 필요할 때만 한 단계 올리세요.

## 동일한 가드레일이 적용된다

호스팅이든 아니든, 자율 에이전트는 여전히 행동을 취합니다. **최소 권한**, **제한된 비용/반복 횟수**, **위험한 단계에 대한 사람의 승인**을 유지하세요 — [에이전트 보안](/docs/security/securing-agents)과 [자율 실행 강화](/docs/security/hardening-autonomous-runs)를 참고하세요.
