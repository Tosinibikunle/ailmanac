**Output styles** change *how* Claude Code communicates — its tone, verbosity, and whether it explains its reasoning — without changing *what* it can do. Think of it as a personality/format preset for responses.

## Why use them

- **Match your workflow.** Terse and action-first when you're moving fast; explanatory when you're learning.
- **Teaching mode.** Some styles add educational commentary — great while you're getting to know a codebase or a technique.
- **Consistency.** Set a preferred style so every session reads the way you like.

## Common styles

Typical built-in styles range from a concise default to more **explanatory** or **learning-oriented** modes that walk through the "why." Set the active style via the relevant command/setting (see the [official docs](https://docs.anthropic.com/en/docs/claude-code/output-styles) for the current list and exact mechanism).

## Output styles vs CLAUDE.md vs response styles

- **Output style** = presentation preset for Claude Code's replies (tone/verbosity).
- **[CLAUDE.md](/docs/claude-code/claude-md)** = persistent *project* facts and rules (what to do, conventions, guardrails).
- **[Custom instructions / response styles](/docs/claude-app/custom-instructions)** = the Claude.ai consumer-app cousin.

:::tip Use the right lever
Want Claude to "always run tests before finishing"? That's a **CLAUDE.md guardrail**, not a style. Want it to "explain each step as you go"? That's an **output style**. Keep behaviour rules in CLAUDE.md and presentation in styles.
:::
